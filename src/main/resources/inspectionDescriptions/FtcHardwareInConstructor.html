Reports use of hardwareMap inside an OpMode constructor.

Hardware access must happen after the OpMode is initialized by the SDK, not during object construction.

Why it matters: During construction, hardwareMap may be null or not yet configured; accessing it can crash or behave unpredictably.

Triggered when: A constructor of a class recognized as an OpMode reads hardwareMap (for example, hardwareMap.get(...) or hardwareMap.something).

Quick fix: Delete statement removes the entire offending statement. Move hardware initialization into runOpMode() (LinearOpMode) or init() (Iterative OpMode).

Bad:

public class Drive extends LinearOpMode {
    DcMotor lf;
    public Drive() {
        lf = hardwareMap.get(DcMotor.class, "lf"); // not safe here
    }
    public void runOpMode() { /* ... */ }
}

Good:

public class Drive extends LinearOpMode {
    DcMotor lf;
    @Override public void runOpMode() {
        waitForStart();
        lf = hardwareMap.get(DcMotor.class, "lf");
        while (opModeIsActive()) {
            // ...
        }
    }
}
