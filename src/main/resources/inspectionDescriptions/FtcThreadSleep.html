Reports blocking sleep/wait calls in FTC OpModes such as Thread.sleep(...), TimeUnit.sleep(...), SystemClock.sleep(...), Object.wait(...), or LinearOpMode.sleep(...).

Blocking calls stall the event loop and can freeze telemetry and responsiveness; remove them or use a non-blocking loop that checks opModeIsActive().

Why it matters: The FTC runtime expects OpModes to return to the loop quickly. Blocking the thread can lead to watchdog warnings and missed updates.

Triggered when: A call to any of the above methods appears inside a class recognized as an OpMode in team code.

Quick fix: Delete blocking call removes the entire statement.

Bad:

public void runOpMode() {
    waitForStart();
    Thread.sleep(500); // blocks the event loop
}

Good:

import com.qualcomm.robotcore.util.ElapsedTime;

public void runOpMode() {
    waitForStart();
    ElapsedTime t = new ElapsedTime();
    while (opModeIsActive() && t.milliseconds() < 500) {
        // do lightweight work
        idle();
    }
}
