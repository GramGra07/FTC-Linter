Reports when multiple TeamCode packages or source roots are detected in the project, which can lead to duplicated OpModes, conflicting classpaths, and unpredictable FTC behavior.

This inspection expects a single TeamCode module and package in an FTC project. It flags cases such as:

  • More than one module name contains “TeamCode”.
  • Multiple source roots or directories contain a TeamCode segment (e.g., /TeamCode/ or \TeamCode\).
  • More than one Java/Kotlin package contains teamcode (for example, com.myteam.teamcode and org.other.teamcode).

Why it matters

  • OpModes may be discovered twice or not at all depending on classpath order.
  • Build and IDE tooling can index/compile unintended sources.
  • Refactors across multiple TeamCode packages are error‑prone.

How to fix

  • Keep exactly one TeamCode module and one teamcode package in the project.
  • Move or delete stray TeamCode folders and consolidate code into a single package (for example, com.myteam.teamcode).
  • Update project/module names so that only the intended module contains “TeamCode”.

Examples

// Two TeamCode packages in the same project (will be flagged)
package com.myteam.teamcode;    // TeamCode #1
public class DriveTeleOp {}

package org.other.teamcode;    // TeamCode #2
public class AutoBlue {}

Configuration

  • Enabled by default under FTC Linter inspections.
  • Settings: Settings/Preferences | Editor | Inspections | FTC Linter | Multiple TeamCode packages detected.

Notes

  • Detection uses module names, directory paths, and package names containing “TeamCode”/“teamcode”.
